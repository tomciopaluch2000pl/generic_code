## 10. Recovery branch – intended execution sequence

This is the technically correct sequence that should be followed from the validated lab state.

### 10.1 Identify source backup images

Validated images available:

- offline full: `20260217093450`
- online full: `20260217102911`
- online incremental: `20260217103901`

Representative history query:

```bash
db2 list history backup all for KSLAB1
```

Important syntax note:

- use `LIST HISTORY BACKUP ...`, not `LIST HISTORY RESTORE ...`

### 10.2 Start tablespace restore into staging DB

Recovery objective:

- restore source tablespace `TS2` into `KSL1STG`

Representative syntax pattern:

```bash
db2 restore db KSLAB1 \
  tablespace (TS2) \
  from /ars/home/kslab/backup/full/tgt1,/ars/home/kslab/backup/full/tgt2 \
  taken at 20260217102911 \
  into KSL1STG \
  redirect
```

Important syntax finding from the lab:

- `FROM` / `TAKEN AT` must appear before `INTO`

### 10.3 Set staging containers after restore enters redirected state

Representative syntax pattern:

```bash
db2 set tablespace containers for 5 using (
  file '/ars/home/kslab/data/KSL1STG/ts2_cont1' <size>
)
```

Important validated lesson:

- do not run this too early,
- an attempted `SET TABLESPACE CONTAINERS FOR 5` failed with `SQL1008C` because tablespace `TS2` was not yet present in the staging recovery state.

### 10.4 Continue restore

Representative syntax pattern:

```bash
db2 restore db KSLAB1 continue
```

### 10.5 Apply incremental / logs and roll forward to PIT

Representative syntax pattern:

```bash
db2 rollforward db KSL1STG to 2026-02-17-10.47.03 and stop
```

Operational note:

- in the final validated rerun, verify whether additional syntax nuance is required for tablespace-level vs DB-level rollforward in the chosen restore method.
- because the source online full and incremental were taken with `INCLUDE LOGS`, required recovery logs should be more straightforward to access.

### 10.6 Validate recovered row in staging

Representative query:

```sql
SELECT * FROM T_TS2 WHERE ID = 12
```

Expected result after successful PIT recovery:

- row `ID=12` is present again in `KSL1STG`

### 10.7 Export recovered data

Representative syntax pattern:

```bash
db2 "export to /tmp/t_ts2_id12.del of del \
  select * from T_TS2 where ID = 12"
```

### 10.8 Re-import into source DB

Representative syntax pattern:

```bash
db2 "import from /tmp/t_ts2_id12.del of del insert into T_TS2"
```

Post-check:

```sql
SELECT * FROM T_TS2 ORDER BY ID
```

Expected restored logical result in source:

- `10`
- `11`
- `12`
- `13`

## 11. Errors and lessons learned

### 11.1 Wrong restore clause order

Observed issue:

- restore failed when `INTO` was placed before `FROM`

Correction:

- use `FROM` / `TAKEN AT` first, then `INTO`

### 11.2 Wrong catalog column assumptions

Observed issue:

- attempted use of wrong column names such as `TBSP_ID` / `TBSP_NAME`

Correction:

- use `TBSPACEID` and `TBSPACE`

### 11.3 Wrong LIST HISTORY assumption

Observed issue:

- attempted `LIST HISTORY RESTORE`

Correction:

- use `LIST HISTORY BACKUP` or `LIST HISTORY ROLLFORWARD`

### 11.4 Early container redirection attempt

Observed issue:

- `SET TABLESPACE CONTAINERS FOR 5` returned `SQL1008C`

Interpretation:

- the target tablespace had not yet entered the staging restore flow and therefore did not exist in the staging catalog at that moment.
