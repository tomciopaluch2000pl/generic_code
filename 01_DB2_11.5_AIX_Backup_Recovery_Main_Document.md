# DB2 11.5 on AIX 7.2 – Backup & Recovery Main Document

## 1. Purpose

This document describes the design, execution approach, and validated outcomes of a DB2 11.5 backup and recovery lab executed on AIX 7.2. The lab was built to validate practical backup and recovery procedures that can later be scaled and industrialized for larger environments, including a future production-oriented landscape in the ~400 TB range.

The primary goal was not only to execute backups, but to verify a realistic recovery model for operational incidents, especially a **logical data loss affecting a limited subset of objects**. The preferred operational model is to avoid unnecessary restoration of the full production database and instead recover only the required data through a **staging database recovery pattern**.

---

## 2. Lab objectives

The lab was designed to validate the following areas:

1. Enable and verify archive logging required for rollforward recovery.
2. Establish a clean baseline using an offline full backup.
3. Execute a performance-oriented online full backup.
4. Execute an online incremental backup with log inclusion.
5. Simulate application data changes after backup.
6. Capture a point-in-time marker before an unwanted DELETE.
7. Validate the recovery design for restoring only the affected tablespace into a separate staging database.
8. Roll forward the recovered object to a timestamp just before the data loss.
9. Export the recovered rows and import them back into the source database.
10. Capture syntax issues and operational pitfalls that matter in real recovery situations.

---

## 3. High-level recovery strategy

### 3.1 Why a staging database was used

For a logical incident such as an accidental `DELETE`, restoring the full source database is usually too disruptive. The lab therefore followed a more production-relevant pattern:

- keep the source database online and untouched,
- restore only the affected tablespace,
- perform the restore into a separate **staging database**,
- roll the staging copy forward to the required point in time,
- extract only the missing or damaged business rows,
- reintroduce those rows into the original source database.

This pattern is especially important when:

- the source database is large,
- outage windows are limited,
- only a small object subset is affected,
- business wants minimal disruption.

### 3.2 Recovery scenario validated in principle

The incident simulated in the lab was the deletion of a row in a table located in tablespace `TS2`. The intended recovery pattern was:

1. identify the point in time immediately before the DELETE,
2. restore `TS2` into the staging database,
3. apply rollforward to that point in time,
4. query the recovered row,
5. export/import the row back to the source.

This is the core design pattern that should be retained for production use.

## 7. Incident simulation

After the incremental backup, additional rows were inserted:

- `T_TS1`: inserted row with `ID = 4`
- `T_TS2`: inserted row with `ID = 13`
- `T_TS3`: inserted row with `ID = 23`

A point-in-time marker was captured before the destructive action:

- **PIT before DELETE:** `2026-02-17-10.47.03`

Then a logical data loss was introduced:

- row `ID = 12` was deleted from `T_TS2`,
- the transaction was committed.

State observed afterward in `T_TS2`:

- rows present: `10`, `11`, `13`

This created the exact recovery use case the lab was designed to solve.

## 9. Recovery design pattern for the incident

The intended recovery pattern for the deleted row in `TS2` is:

1. restore the source tablespace `TS2` into `KSL1STG`,
2. use redirected restore if container remapping is needed,
3. complete restore processing,
4. roll forward the staging database or restored object to `2026-02-17-10.47.03`,
5. query `T_TS2` in staging and verify row `ID=12` exists,
6. export the row,
7. import or merge it back into `KSLAB1`.

This pattern remains the preferred production recovery method for limited logical corruption or row loss.

## 10. Errors and syntax findings captured during the lab

The lab provided several useful operational findings.

### 10.1 RESTORE syntax clause order matters

A restore attempt failed because `INTO` was placed before `FROM`.

Key learning:

- the correct DB2 syntax order requires `FROM` / `TAKEN AT` before `INTO`.

This is important under pressure, because restore syntax mistakes are common during incident recovery.

### 10.2 Catalog column naming trap

In `SYSCAT.TABLESPACES`, the correct columns observed were:

- `TBSPACEID`
- `TBSPACE`

Not:

- `TBSP_ID`
- `TBSP_NAME`

This matters when scripting tablespace discovery and recovery automation.

### 10.3 LIST HISTORY syntax

A syntax trap was observed with `LIST HISTORY`.

Correct pattern:

- `LIST HISTORY BACKUP ...`
- `LIST HISTORY ROLLFORWARD ...`

There is **no `RESTORE` history type** in that syntax form.

### 10.4 SQL1008C during SET TABLESPACE CONTAINERS

Attempting to run `SET TABLESPACE CONTAINERS FOR 5` in `KSL1STG` failed with `SQL1008C` because the target tablespace did not yet exist in the staging DB.

This confirmed an important operational truth:

- container redirection can only proceed after the restore process has actually introduced the target tablespace definition into the staging recovery flow.

### 10.5 Staging DB initial tablespace inventory

Immediately after `CREATE DB`, only default tablespaces existed in `KSL1STG`:

- `0 SYSCATSPACE`
- `1 TEMPSPACE1`
- `2 USERSPACE1`
- `3 SYSTOOLSPACE`

This confirmed that user tablespaces `TS1`, `TS2`, `TS3` had not yet been brought in by the restore branch at that stage.

## 12. Production design considerations for large environments

The same recovery model becomes even more important for large databases such as ~400 TB environments.

### 12.1 Why full DB restore is usually not acceptable at large scale

At multi-hundred-terabyte size, a full restore is often:

- operationally too slow,
- disruptive to multiple applications,
- storage-intensive,
- difficult to fit into recovery windows.

Therefore, recovery design should favor:

- object-level or tablespace-level recovery whenever possible,
- isolated staging recovery patterns,
- controlled export/import reintegration,
- strong backup chain discipline,
- deterministic log retention.

### 12.2 Backup design recommendations for ~400 TB target state

For a future production-scale implementation, the following should be formalized:

- backup target parallelism sized to available I/O channels,
- multiple backup mount points or storage stripes,
- compression validation based on CPU headroom,
- clear separation of full and incremental backup paths,
- documented retention of archive logs required for PIT recovery,
- routine recovery drills, not just backup success checks,
- verification of catalog queries and restore automation scripts.

### 12.3 Recovery model recommendations

Recommended production pattern:

- use online full backup as chain anchor,
- use incremental strategy where change rate justifies it,
- retain logs consistently,
- use isolated staging DB for logical incidents,
- predefine restore container mapping standards,
- document recovery runbooks with exact syntax order,
- test row-level extraction procedures in advance.

### 12.4 Operational readiness requirement

For large-scale estates, the success criterion is not “backup completed” but rather:

- can a specific business object be recovered within the agreed RTO/RPO?

That is the real architectural value of this lab.

## 14. Conclusion

The lab successfully validated the core building blocks required for a production-grade DB2 backup and recovery model on AIX 7.2:

- archive logging,
- offline and online backup chain handling,
- incremental backup readiness,
- logical incident simulation,
- staging-based recovery design,
- identification of high-risk syntax and recovery pitfalls.

The most important architectural outcome is the confirmation that the **right recovery model for limited logical loss is not a full database restore, but a targeted tablespace recovery into a separate staging database followed by selective data reintegration**.

That principle is directly reusable for larger and more critical environments.
