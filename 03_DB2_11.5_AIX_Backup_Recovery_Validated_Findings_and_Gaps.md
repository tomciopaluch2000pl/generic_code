# DB2 11.5 on AIX 7.2 – Validated Findings, Gaps, and Recommendations

## 1. Executive summary

The lab successfully validated the DB2 backup chain and the core recovery architecture for handling a limited logical data loss event without restoring the entire production database. The most important validated design outcome is the use of a **separate staging database** for targeted recovery.

At the same time, the lab also showed that DB2 restore/recovery syntax is operationally sensitive, especially under pressure. Several errors were discovered early, which is valuable because these are exactly the types of mistakes that would delay a real incident recovery.

---

## 2. What was validated with strong confidence

### 2.1 Backup and logging readiness

Confirmed:

- archive logging was enabled,
- active and archive log paths were separated,
- `TRACKMOD YES` was enabled,
- baseline offline full backup completed successfully,
- online full backup completed successfully,
- online incremental backup completed successfully.

### 2.2 Logical recovery scenario setup

Confirmed:

- isolated tablespaces `TS1`, `TS2`, `TS3` were created,
- user tables were distributed across those tablespaces,
- a valid incident was simulated by deleting `ID=12` from `T_TS2`,
- a precise PIT timestamp was captured before the DELETE: `2026-02-17-10.47.03`.

### 2.3 Staging recovery model

Confirmed:

- dedicated staging database `KSL1STG` was created,
- logging configuration was applied there as well,
- backup pending was cleared using backup timestamp `20260217122605`.

## 3. Most useful technical findings

### 3.1 Restore syntax order is non-trivial

Finding:

- DB2 restore syntax is unforgiving with clause order.

Observed problem:

- placing `INTO` before `FROM` caused the restore to fail.

Recommendation:

- freeze exact restore syntax in the final runbook and do not improvise during incidents.

### 3.2 Catalog queries must use verified column names

Finding:

- assumptions about catalog column names can be wrong.

Observed correction:

- use `TBSPACEID` and `TBSPACE` in `SYSCAT.TABLESPACES`.

Recommendation:

- all automation should be based on validated catalog queries, not remembered column names.

### 3.3 LIST HISTORY syntax must be exact

Finding:

- restore history was initially approached with invalid syntax.

Observed correction:

- use `LIST HISTORY BACKUP ...` or `LIST HISTORY ROLLFORWARD ...`

Recommendation:

- include canonical history commands directly in the incident runbook.

### 3.4 Container redirection depends on restore state

Finding:

- container remapping cannot be performed before the target tablespace exists in the staging recovery workflow.

Observed symptom:

- `SQL1008C` during `SET TABLESPACE CONTAINERS FOR 5`.

Recommendation:

- make redirected restore state transitions explicit in the procedure.

---

## 4. Current gap

The only meaningful execution gap is the last part of the object recovery flow:

- redirected restore of `TS2` into `KSL1STG`,
- rollforward to the PIT,
- export/import of the recovered row.

This is not a design gap. It is an **evidence gap** in the captured run.

That distinction is important:

- the recovery method is correct,
- the final proof needs one clean rerun and evidence capture.

---

## 5. Why this lab is architecturally relevant

For large systems, including a future ~400 TB DB2 estate, the recovery challenge is usually not “can we take backups?” but rather:

- can we recover a small but business-critical data subset without restoring the whole database?

This lab directly addresses that question.

The recovered-object-through-staging approach is the right architectural direction because it:

- limits production disruption,
- reduces recovery blast radius,
- supports targeted remediation,
- can be standardized into repeatable runbooks.
